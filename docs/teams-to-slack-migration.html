Microsoft Teams 채팅　→　Slack

사내 협업도구 전환 검토 보고서

검토자 지나 (검증형 기획자 관점)　·　작성일 2026-06-18
대상 사내 채팅 도구를 Microsoft Teams → Slack 으로 전환하는 IT/운영 건
방식 양 플랫폼 공식 문서 조사 + 리스크 분석
한 줄 결론

과거 대화를 그대로(빠짐없이) 옮기는 "공식 정품 경로"는 없습니다.

전환 자체는 가능합니다. 다만 채팅 기록·파일·답글·이모지까지 100% 보존하려면 외부 유료 업체에 전적으로 의존해야 하고, 그마저도 우리가 가려는 방향(Teams→Slack)이 시장에서 흔치 않은 방향이라 검증이 먼저 필요합니다.

현실적 권고는 "슬랙에서 새로 시작 + 과거 기록은 Teams에 읽기전용으로 보관" 입니다.

1한눈에 보는 요약

항목결론
Slack 새 업무공간 구축문제 없음 표준 작업
채널 + 채널 메시지 + 사용자 기본 이관부분 가능 공식 파일 업로드 방식 (정보 손실 큼)
1:1 / 그룹 개인 대화(DM) 이관공식 경로 없음 외부 업체 필요
파일·답글·이모지·멘션 보존공식 미지원 외부 업체 의존
Teams에서 데이터 빼내기가능 단 라이선스·권한 요건 있음
가장 큰 리스크시장 도구 대부분이 반대 방향(Slack→Teams)에 맞춰져 있음
최종 권고새로 시작 + 과거는 Teams 보관. 전체 이관은 비용·리스크 대비 효용 낮음

2두 플랫폼은 각각 무엇을 지원하나

쉽게 말하면 — Teams는 "데이터를 밖으로 꺼내주는 것"까지만 합니다. Slack은 "정해진 형식의 파일만 받아주는 것"까지만 합니다. 문제는 Teams가 꺼내주는 형식과 Slack이 받아주는 형식이 서로 맞지 않는다는 데 있습니다.
▸ Teams 측 — 데이터 꺼내기
  • 방법 A. eDiscovery
    채팅·채널 메시지를 참가자·시간 포함해 추출. 원래 법무·감사용. 상위 라이선스(E5급) 필요. 결과물은 Slack이 바로 못 읽는 형식.
  • 방법 B. Graph Export API
    개발 방식으로 메시지를 자동 추출. 관리자 승인·기술 인력 필요. 2025-08-25부로 메시지당 과금은 폐지됐으나 승인 절차는 남음.
▸ Slack 측 — 데이터 받기
  • 공식으로 받아주는 소스는 딱 둘:
    ① 다른 Slack 업무공간　② CSV/텍스트 파일
  • Teams를 직접 받아오는 경로는 없음 (공식 문서 확인됨)
  • CSV로 받을 수 있는 건 채널·메시지 본문·사용자 계정뿐. 개인대화·파일·답글·이모지·비공개 채널·프로필은 전부 못 받음.

외부 마이그레이션 업체에 대한 결정적 주의

시장에는 CloudFuze·Cloudiway·ShareGate·AvePoint 같은 전문 업체가 있습니다. 그러나 이들은 대부분 "Slack에서 Teams로" 옮기는 방향에 최적화돼 있습니다. 우리는 정반대 방향입니다.
  • Cloudiway — 공식 페이지상 출발지로 Slack만 지원. Teams→Slack은 미지원으로 확인됨.
  • CloudFuze — 양방향을 시사하는 문구는 있으나 Teams→Slack을 명시 보증하는 공식 페이지는 확인 안 됨 → 확인 필요 항목.
  • 업체가 기술적으로 처리할 수는 있어도, 검증된 사례·문서가 빈약하므로 계약 전 시범 이관(PoC)으로 반드시 검증해야 합니다.

4리스크 — 먼저 풀어야 할 것 vs 관리하면 되는 것

블로커 1. 방향성 미검증 (최우선)

Teams→Slack은 시장 도구의 역방향입니다. Cloudiway는 명시적 미지원, CloudFuze 등은 공식 보증 미확인. "전체 기록 자동 이관"을 목표로 확정하려면 그 전에 업체 시범 이관으로 방향 지원·보존율을 실측해야 합니다. 이 검증 없이는 일정·예산 확정 불가.

블로커 2. 무손실 공식 경로 없음

Slack 공식 방식만으로는 개인대화·파일·답글·이모지가 전부 사라집니다. "과거 대화를 Slack 안에서 그대로 본다"가 필수라면 외부 업체 비용·종속이 전제가 됩니다.

우려 데이터 정합성

CSV 방식은 채널·작성자가 함께 옮겨지지 않은 메시지는 누락됩니다. 부분 이관 시 대화 맥락이 깨질 수 있어 이관 후 검수가 필수입니다.

우려 개인 대화의 사생활·법적 이슈

1:1 DM은 개인 대화입니다. 일괄 이관은 개인정보 접근 권한을 동반하므로 한국 개인정보보호법·노무 관점에서 법무/인사 사전 합의가 필요합니다.

우려 라이선스·비용

Teams 추출 API 과금은 폐지됐지만, ① 승인 절차 ② eDiscovery용 E5급 라이선스 보유 여부 ③ 외부 업체 비용(보통 사용자/메시지 단위)이 남습니다.

우려 구조 차이 · 멤버 매핑

Teams는 "팀>채널" 2단 구조, Slack은 평면 구조라 매핑 규칙 설계가 필요합니다. 이메일 기준 사용자 매칭이라 퇴사자·외부 게스트·봇 계정은 누락 위험이 있습니다.

우려 봇·연동·커스텀 이모지 · 병행 운영

봇·연동 앱·커스텀 이모지는 이관 대상이 아니라 Slack에서 새로 구축해야 합니다. 또 전환 당일 100% 이동은 비현실적이라 두 도구를 함께 쓰는 기간의 혼선·강제 전환일 정책이 필요합니다.

5권고

검토자 관점에서 "전체 기록을 Slack으로 무손실 이관"하는 것은 비용·리스크 대비 효용이 낮습니다. 다음을 권고합니다.

1
기본안 — Slack에서 새로 시작 (Clean cutover)

Slack을 새 업무공간으로 만들고 채널 구조만 다시 설계해 출발. 과거 Teams 대화는 읽기전용으로 N개월 보관해 필요할 때 조회. 자주 보는 핵심 채널만 골라 CSV로 텍스트 보관(보조).

2
전체 이관이 꼭 필요하다면 (조건부)

착수 전 업체 시범 이관(PoC) 필수 — Teams를 출발지로 한 방향 지원 여부 + 답글·이모지·파일·DM의 실제 보존율을 샘플로 실측. 라이선스·업체 비용 산정. DM 이관은 개인정보·노무 합의를 선결 조건으로.

3
공통 선행작업

현재 M365 라이선스 등급 확인 · Slack 플랜 결정 · 채널/사용자 매핑표 작성 · 봇·연동·이모지 재구축 목록화 · 병행 운영 기간과 강제 전환일 정책 수립.
